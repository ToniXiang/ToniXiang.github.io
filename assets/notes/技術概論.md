SKIP 是由 Sun Microsystems 提出的密鑰管理協議，用於為 IP 層安全協定（特別是 IPsec）分配與管理加密金鑰
SKIP 的設計特色是 不需要在每次建立連線時重新協商金鑰，而是使用長期公鑰與臨時金鑰結合的方式來產生共享密鑰
後來，IETF（Internet Engineering Task Force）在設計 IPsec 標準時，考慮了多種金鑰管理方案，
包括 SKIP、Photuris 等。最終選擇了結合多種理念發展出的 IKE（Internet Key Exchange），
作為正式的 IPsec 金鑰交換協議

可減少公有 IP（Public IP）位址使用數量

PAT（Port Address Translation）：多台主機共用同一 IP，但用不同 Port 區分
Port Mapping：把「外部的一個連線埠」對應到「內部某台電腦的某個埠」
RDP 遠端桌面：將外部 3389 對應到內部電腦的 3389

死結(deadlock)：若程序持有資源 A，等待資源 B，但另一程序持有 B 等 A，且資源不可剝奪
預防 → 死結(互斥、保持並等待、不可剝奪、循環等待)不會發生，但效率可能低
避免 → 銀行家演算法（Banker’s Algorithm）
偵測與解除 → 資源分配圖 / 等待圖（Wait-for Graph）

黑洞攻擊（Blackhole）：惡意節點宣稱自己有最佳路徑，誘導傳送流量到自己，然後直接丟棄或不轉發封包，破壞可用性
蟲洞攻擊（Wormhole）：攻擊者在網路中建立一條低延遲隧道，把封包從一處「快速搬運」到另一處，能破壞路由選擇與定位協定
分割攻擊（fragmentation evasion）：利用 IP 片段分割來逃避 IDS 或引發錯誤解析

Session Hijacking(會話劫持)：竊取 Cookie 以登入受害者帳號
---
應用層攻擊（Volume-based）：HTTP Flood、Slowloris、SQL Injection、XSS、Remote Code Execution (RCE)
表達層攻擊：SSL/TLS 配置弱點
會議層攻擊：會話劫持、TLS/SSL 中間人
傳輸層攻擊：SYN flood(很多未完成三次握手)、UDP flood(Fraggle)、TCP reset
網路層攻擊：Smurf Attack、ICMP flood Attack、Ping of Death 死亡之 Ping(超大 ICMP 封包，使用目標當機或重啟)、DNS 放大或反射攻擊
資料連接層攻擊：MAC Flooding Attack、ARP 欺騙
實體層攻擊：無線電干擾、纜線剪斷、旁路竊聽
密罐（Honeypot）服務可偵測異常網路活動，正式環境的誘餌，觀察病毒特徵、攻擊手法
---
LAND：封包中來源和目的相同，誘發目標自我回應或錯誤，導致當機
Smurf：透過向廣播發送偽造 ping，讓大量主機回覆同一目標，造成流量放大與 DoS
---
OSI 各層防火牆對照表
L1、L6：無
L2：MAC Filtering 、 Bridge Firewall，依據 MAC 位址過濾封包
L3：Packet Filter Firewall，根據 IP 位址與協定
L4：Stateful Firewall、Packet Filter Firewall，檢查埠號、連接狀態
L5：Session-aware Firewall，確認會話建立、維持狀態
L7：Web Application Firewall(WAF)，阻擋 SQL Injection、XSS、過濾惡意爬蟲或自動化攻擊
---
L3/L4 網路防火牆的訪問控制列表（Access Control List，ACL）
通常只看標頭欄位（無狀態）
明確分 inbound/outbound 就是根據這些欄位設定「允許 / 拒絕」
---
應用了 ACL 概念的防火牆：L3 過濾型網路防火牆（Packet Filtering Firewall），會過濾的封包參數：
[1]來源 IP 位址（Source IP Address）
[2]目的 IP 位址（Destination IP Address）
[3]來源埠號（Source Port Number）
[4]目的埠號（Destination Port Number）
[5]協定類型（Protocol Type，例如 TCP、UDP、ICMP）
[6]封包方向（Inbound / Outbound）
---
白名單（Whitelist）：「允許制」 → 須在名單內才可通過
黑名單（Blacklist）：「禁止制」 → 只要不在名單內就可通過
白名單比黑名單更安全，但不是萬無一失。最佳做法是搭配其他防護機制
例如：入侵偵測 IDS、行為分析、加密與身分驗證
---
SQL Injection（SQL 注入）
類型：
[1]Error-Based SQLi（錯誤回傳型）：攻擊者藉由讓資料庫產生錯誤訊息（例如語法錯誤、型別錯誤），從錯誤內容推測資料庫結構或欄位名稱
[2]Time-Based SQLi（時間盲注 / 盲注）：無明顯回應差異時，透過讓資料庫延遲（例如 SLEEP）判斷布林結果，藉回應時間推斷資料
[3]Union-Base SQLi（聯合查詢型）：利用 UNION SELECT 將攻擊者控制的查詢結果合併至原查詢回傳，以取得資料
[4]Boolean-Based Blind SQLi（布林盲注）：透過回應內容差異（true/false）推斷資料（例如頁面是否回傳特定內容）
防範：
[1]使用參數化查詢 / Prepared Statements（最重要）
[2]最小權限原則：連線使用低權限帳號，避免用 admin 帳號執行應用程式查詢
[3]使用 ORM 或高階資料庫層（例如 Entity Framework、SQLAlchemy、Hibernate）→ 它們預設使用參數化
[4]輸入驗證與白名單：對格式可預期的欄位（數字、email、日期）做強驗證
[5]避免把敏感資訊放在錯誤訊息或日誌中（不要回傳完整 SQL 給使用者）
[6]使用 WAF與 SQL 容錯／監控：偵測異常查詢或注入嘗試
[7]定期掃描與滲透測試（由合法安全團隊執行）
[8]避免靠單純 escaping 作為唯一防線，因為容易繞過
測試時應由合法授權的安全團隊進行滲透測試，避免未授權攻擊
常見攻擊字串變化很多（例如 "' OR 1=1 --、" OR 'a'='a 等），因此以參數化為首要策

FIDO（Fast IDentity Online）是一套由 FIDO Alliance 建立的驗證標準
公開金鑰加密：終端裝置存私鑰；伺服器存對應的公鑰
目的：用密鑰對替代密碼（或與密碼搭配），降低釣魚、重放、伺服器側憑證外洩造成的風險

DevOps：開發（Development）與營運（Operations）的整合，追求 快速、持續交付
DevSecOps：DevOps 多加 安全性(Security)，在開發、測試、部署的每一階段，都自動化地進行安全檢查

測試層次
[1]單元測試(Unit Testing)：最小可測單位（通常是一個函式或類別的方法）做驗證｛開發階段｝
[2]整合測試(Integration Testing)：檢驗多個模組或系統元件如何一起運作｛開發後期｝
[3]系統測試 (System Testing) ｛測試階段｝
[4]驗收測試 (Acceptance Testing) ｛上線前｝
其他
[1]源碼檢測(Source Code Analysis）：靜態測試的一種，不執行程式，利用工具檢查原始碼
[2]模糊測試（Fuzzing）：自動把大量「亂丟」或有策略生成的輸入資料餵給程式
[3]滲透測試(Penetration Testing）：模擬真實攻擊者對系統進行評估，例如：社交工程、密碼破解、弱點掃描
[4]軟體測試(Software Testing）：泛指所有驗證軟體正確性與品質的活動，包含單元、整合、系統與安全性測試
安全性測試：模擬測試+滲透測試
---
密碼潑灑：大量帳號 + 單一弱密碼
暴力破解（Brute-force）：針對單一帳號或小數帳號，嘗試極大量不同密碼
憑證填充（Credential stuffing）：使用從洩露資料庫得到的「帳號+密碼」配對批量登入
